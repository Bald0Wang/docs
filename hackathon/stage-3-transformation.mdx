## 阶段目标

完成从“学生作业”到“职业作品”的包装：让评委一眼看懂价值，让企业一眼看懂能力，让作品能够直接进入简历与面试叙事。

> 关键词：标准化、可复现、可展示、可讲述、可投递

## 1. 通识必修 (全员)

- **职业伦理**：AI 版权、数据隐私与开源协议。
- **结构化表达**：金字塔原理，高效进行 Pitch 汇报。
- **项目复盘**：沉淀方法论，将经历转化为简历上的亮点。

### 合规与边界最小清单
- 是否包含用户隐私/敏感数据？是否脱敏？
- 是否明确标注 AI 生成内容与来源？
- 是否说明模型能力边界与“不能做什么”？
- 是否说明开源依赖与许可（至少列出主要依赖）？

## 2. 专精技能树（分角色任务书）

### 市场方向
- **路演演示**：学习 TED 式演讲，传递商业价值。
- **防御策略**：分析竞争壁垒与护城河逻辑。
- **行业深度**：撰写行业分析白皮书。

#### 路演（Pitch Deck）建议结构（5 分钟版）
1. 你要解决的**问题**是什么（证据 1–2 条）
2. 你的**方案**是什么（一句话 + Demo 截图/流程）
3. 为什么你们的方案**更好**（差异化/护城河）
4. 你们如何证明它**有效**（数据/访谈/可解释指标）
5. **成本与风险**是什么（并给出降级策略）
6. 下一步迭代路线（1–2 周、1–2 月的计划）

### 产品方向
- **作品集包装**：梳理设计背后的思维过程（Case Study）。
- **视频演示**：剪辑产品宣传视频。
- **像素验收**：进行 Pixel Perfect 级别的最终验收。

#### Case Study（作品集）建议结构
- 背景与目标：用户是谁、场景是什么、成功怎么衡量
- 研究与洞察：证据链（访谈/数据/竞品）
- 方案探索：至少 2 个方案的权衡（为什么不选另一个）
- 关键交互：主链路 + 异常流 + 可信体验设计
- 验证与迭代：可用性测试结果 + 改版记录
- 最终结果：指标/反馈 + 下一步路线图

### 技术方向
- **开源规范**：维护高质量 README 与 API 文档。
- **代码重构**：清理代码，优化可维护性。
- **技术博客**：记录架构选择与技术决策。

#### README（开源级）必须包含的章节
- 项目简介（价值一句话 + 适用场景）
- 功能列表（按主链路描述，不堆功能）
- 快速开始（最少步骤、可复现）
- 架构说明（数据流/组件职责/关键依赖）
- AI 说明（模型选择、Prompt 规范、评测/回归方式）
- 合规与边界（隐私、风险、免责声明）
- 贡献与许可（License/致谢）

## 3. Milestone 3：最终验收与交付（Final Delivery）

### 验收原则
- **市场作为验收官（UAT）**：对照最初的需求与成功指标验收
- **产品作为体验官**：主链路是否顺畅，异常流是否可用
- **技术作为稳定官**：部署是否稳定、是否可复现、是否有回归记录

### 终稿交付包（团队级）
- 路演 PPT + 演示脚本（3–5 分钟）
- 可访问 Demo（或可复现运行方式）
- 统一 README（背景/功能/使用/边界/合规）
- 证据附件（访谈纪要/数据截图/评测用例与回归记录）

### 简历项目描述（写法模板）
- **一句话**：我做了一个什么系统，解决了谁在什么场景下的什么问题
- **我的职责**：在三人 MCU 中我负责什么（可量化）
- **关键结果**：用数据/反馈/评测证明有效（越具体越好）
- **关键难点与解决**：风险是什么，怎么权衡与降级
