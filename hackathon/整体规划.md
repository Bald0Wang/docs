# Datawhale × 魔搭社区：C9 高校黑客松人才培养课程（长期备赛版）

> 角色：你现在是我的顾问，辅助我进行课程设计  
> 核心目标：通过“真实有意义”的黑客松学习旅程，让同学在跨学科协作中持续突破自己；完赛后具备企业需要的能力，可将作品与能力沉淀为可投递、可面试的职业化成果。  
> 核心比喻：黑客松是学生的练功房，修炼成企业需要“骨骼惊奇”的新同学。

---

## 1. 课程愿景与合作背景

我们期待通过本次黑客松为同学们带来一次真实有意义、能提升自身价值、提升团队协作的学习旅程。我们希望覆盖各个学科、各个水平能力的同学，在比赛过程中不断突破自己。

- **Datawhale 的定位**：作为桥梁，打通学校与企业的“最后一公里”
- **魔搭社区（ModelScope）**：提供模型、工具与生态资源，降低 AI 应用开发门槛
- **学校与企业与我们**：汇集高校人才，做企业人才发展的预备课；与企业共创赛题与评审，将人才落位到企业岗位

## 2. 核心培养模型：最小战斗单元（MCU）

### 2.1 为什么要三人组队？

简单把三个人凑在一起不叫团队，叫“团伙”。要让他们产生化学反应，必须定义清晰的输入（Input）与输出（Output），并设置“互锁环节”。

三人组是黑客松的**最小战斗单元**，分别对应企业真实协作链条：

- **市场**：确定方向与价值，负责“为什么做、做得值不值”
- **产品**：把价值变成可用方案，负责“怎么让用户用起来”
- **技术**：把方案变成可运行系统，负责“能不能做、怎么做得稳”

### 2.2 角色职责与对标岗位（建议放入《角色任务书》）

| 角色 (Role) | 核心职责 (Core Responsibility) | 对内交付物 (Internal Deliverables) | 企业对标岗位 |
| --- | --- | --- | --- |
| **市场 (Market)** *(文科类)* | **方向官 & 体验官**：回答“为什么做”“做得怎么样” | 1) 竞品/替代方案分析 2) 用户画像与痛点证据 3) UAT 验收测试报告 | 产品运营、市场分析、项目助理、商业分析 |
| **产品 (Product)** *(设计类)* | **翻译官 & 建筑师**：把需求翻译成界面/流程/AI 交互 | 1) 交互原型（Figma/MasterGo）2) 用户旅程地图 3) AI 交互与提示词体系 | UI/UX、产品经理、AI 交互设计 |
| **技术 (Tech)** *(理工类)* | **实现官 & 边界官**：回答“怎么做”“能不能做” | 1) 可运行代码仓库 2) 技术架构图 3) 技术可行性评估与成本测算 | 全栈/后端/算法应用、研发效能 |

### 2.3 强制融合（互锁）机制（建议作为评分项）

- **技术必须依据产品原型开发**：原型/DRD 不完整不得开工
- **产品必须基于市场证据设计**：没有用户证据/竞品证据不得立项
- **市场必须对成品做验收**：不仅找 bug，更找“体验断点”和“价值偏差”

## 3. 课程结构：三阶段路线图（长期备赛/学期制）

我们将课程分为三阶段：**装备期（赛前）→ 深耕期（赛中/长期迭代）→ 蜕变期（赛后转化）**  
同时引入两条学习主线：

- **通识模块（全员）**：AI 认知、协作方法、表达与职业素养
- **专精模块（分角色）**：市场/产品/技术的技能树与成长路径

> 关键调整：市场与产品方向的学习内容要更强、更“可操作”；并且 AI 技能必须贯穿三阶段持续跟进，而不是只在赛前“扫盲”。

---

## 4. 持续 AI 能力主线（全员贯穿 3 阶段）

### 4.1 AI 能力分层（全员）

- **A0（会用）**：能用 AI 提效（总结、改写、生成草稿）
- **A1（会问）**：掌握提示词结构与迭代方法，知道如何约束输出
- **A2（会评）**：能评估 AI 输出（可靠性、偏差、可用性），会做对照实验
- **A3（会协作）**：把 AI 融入流程（从需求到设计到交付），形成团队 SOP

### 4.2 AI 持续学习机制（建议每周固定节奏）

- **AI Weekly（每周 30-45 分钟）**：本周模型/工具更新、案例拆解、最佳实践复盘
- **Prompt Library（提示词库）**：团队共享可复用的 prompt 模板（研究/设计/文案/评测）
- **AI Output QA（输出质检）**：每周随机抽取 1 份 AI 生成内容做“证据链检查”
- **安全与合规常态化**：隐私、数据授权、开源协议、AI 生成内容版权标注

## 5. 第一阶段：装备期（预备课，赛前 1-2 周）

### 5.1 阶段目标

扫盲、磨合、工具统一，解决“这也是我们需要做的吗？”的认知偏差。

### 5.2 通识必修（全员）

- **AI 认知对齐**：LLM 能做什么/不能做什么；常见幻觉与偏差
- **协作工具链**
  - 文档协作：Feishu/Notion（统一模板与目录）
  - 项目管理：Trello/Jira 看板（任务拆解与节奏）
  - 沟通礼仪：Stand-up（每日站会）、评审会、复盘会
- **最小可行性原则（MVP）**：学会砍需求、做可交付闭环

### 5.3 分角色专精技能树（强化版：市场 & 产品重点加强）

#### 市场方向（装备期技能树）

**关键词：洞察与定义 + 证据链 + AI 辅助研究**

- **痛点挖掘**：区分伪需求与真痛点（5W1H、JTBD）
- **竞品/替代方案侦察**：竞品矩阵、差异化定位
- **证据链构建**：每个结论必须有证据（访谈摘录/问卷数据/公开资料）
- **AI 辅助研究（基础）**
  - 用 AI 做信息聚合与结构化（但必须标注来源与不确定性）
  - 提示词基础：让 AI 扮演“商业分析师/行业研究员/用户研究员”
- **阶段产出物**
  - 《问题定义一页纸》（Problem Statement）
  - 《竞品与替代方案矩阵》
  - 《用户画像 v0 + 关键证据》

#### 产品方向（装备期技能树）

**关键词：具象与交互 + AI 交互范式 + 可用性基础**

- **用户旅程地图（User Journey Map）**：从痛点到触点到情绪曲线
- **原型工具协作**：Figma/MasterGo（组件、协作、评论、版本）
- **交互范式**：CUI（对话）vs GUI（界面）vs 多模态（图片/语音/文档）
- **AI 交互基础**
  - “指令式 → 对话式 → 任务式 Agent”三种产品形态
  - 提示词基础：角色、目标、约束、格式、示例、拒答策略
- **可用性基础**：可发现性、反馈、容错、空状态/异常状态
- **阶段产出物**
  - 《低保真线框图（核心流程）》
  - 《核心交互稿 v0（含异常/空状态）》
  - 《AI 交互说明 v0（输入/输出/约束/拒答）》

#### 技术方向（装备期技能树）

**关键词：开发栈统一 + 快速原型 + AI 应用基础 + 工程协作**

- **开发基础与协作规范**
  - Git 工作流：分支策略、Pull Request、Code Review（禁止“发代码压缩包”）
  - Issue/看板：把需求拆成任务，明确验收标准（Definition of Done）
- **快速原型能力（Demo First）**
  - 快速搭建：Streamlit/Gradio/简单前后端（能在 1-2 天内出可用 Demo）
  - 基础部署意识：本地可跑、他人可复现（README、环境说明）
- **LLM 应用基础（偏“会用会集成”）**
  - API 调用：理解鉴权、限流、超时、重试、费用与成本
  - Prompt 基础：把产品的交互约束落到“输入/输出格式协议”
  - 安全基础：敏感信息脱敏、日志不落隐私、避免提示词注入（Prompt Injection）入门
- **阶段产出物**
  - 《项目脚手架仓库 v0》（可运行 + README + 基础目录结构）
  - 《技术可行性评估 v0》（成本/风险/时间）
  - 《最小可运行 Demo v0》（可被市场/产品同学独立启动体验）

## 6. 第二阶段：深耕期 —— 产品孵化与工程化（4-8 周）

### 6.1 阶段目标

将脆弱 Demo 打磨为 Robust Product（健壮产品）。  
关键词：数据驱动、系统设计、敏捷迭代、持续 AI 能力升级。

### 6.2 通识必修（全员）

- **敏捷管理进阶（Scrum）**：Sprint Planning / Review / Retro
- **数据思维**：北极星指标（North Star Metric）、漏斗、留存、满意度
- **评测意识**：不仅“能跑”，还要“好用、稳定、可解释”
- **AI 协作力进阶**：从“用工具”变成“有流程”（提示词库、评测表、复盘机制）

### 6.3 分角色深度技能树（市场 & 产品强化版）

#### 6.3.1 市场方向（深耕期技能树：商业验证与增长）

**核心定位：从“方向官”升级为“商业验证官”**

- **定量研究（强化）**
  - 问卷设计：避免诱导性提问、量表与样本偏差
  - 数据分析：Excel/飞书表格基础统计；会做交叉分析与结论归因
  - 产出：可信的“需求强度”与“付费/使用意愿”结论
- **商业模式画布（BMC）与定价**
  - 成本结构、收入流、渠道通路、关键资源
  - 竞品定价与价值锚点（Value-based Pricing）
- **冷启动与增长实验**
  - 前 100 用户获取：内容营销、社群、校园渠道
  - 增长实验设计：假设-实验-指标-结论（最小成本验证）
- **AI 持续学习（市场向）**
  - AI 做研究：信息聚合、访谈纪要、主题归纳（但必须做证据链校验）
  - AI 做增长：文案生成/多版本 A/B、渠道内容脚本
  - AI 风险：虚构数据、伪引用、夸大结论的识别与纠偏
- **阶段产出物（必须可审计）**
  - 《商业验证报告》（含方法、样本、数据、结论、风险）
  - 《增长实验日志》（每周至少 1 次实验）
  - 《运营 SOP》（拉新/转化/留存的可执行步骤）

#### 6.3.2 产品方向（深耕期技能树：系统设计与体验闭环）

**核心定位：从“建筑师”升级为“体验总工 + AI 交互负责人”**

- **设计系统（Design System）**
  - 色彩/字体/栅格/组件规范；组件复用与一致性
  - 与开发对齐：命名、状态、组件 API 思维（即使不写代码也要会描述）
- **交互逻辑进阶（重点）**
  - 异常状态：断网、超时、无权限、模型拒答、模型瞎编
  - 可信体验：引用来源展示、置信度提示、可撤销与可追溯
  - 任务拆解：把复杂任务拆成可控步骤（任务式交互）
- **可用性测试与数据闭环**
  - 可用性测试脚本、任务完成率、时间、满意度
  - 小规模 A/B：两套交互方案用数据做决策
- **AI 产品与提示词体系（强化）**
  - Prompt Engineering 进阶：模板化、可版本化、可评测
  - 输出格式协议：结构化 JSON/表格/步骤清单，便于前端渲染与后端处理
  - 人设与语气：品牌一致性、敏感内容处理、拒答与引导话术
  - 提示词调优方法：失败案例库、对照实验、回归测试（prompt regression）
- **阶段产出物**
  - 《UI Kit / 组件库 v1》
  - 《DRD 交互说明文档》（含状态机/异常流/可访问性）
  - 《AI 交互与提示词规范 v1》（含评测用例与回归表）
  - 《可用性测试报告》（含录屏/问题清单/修复优先级）

#### 6.3.3 技术方向（深耕期技能树：工程化落地与质量保障）

**核心定位：从“能跑 Demo”升级为“可交付系统负责人”**

- **LLM 应用开发进阶（重点）**
  - RAG（检索增强生成）：数据准备、向量检索、召回与重排、引用与可追溯
  - Prompt 编排：多步链路、工具调用、函数/工具选择策略
  - 评测与回归：建立用例集（happy path + failure cases），每次改 prompt/代码都要回归
- **系统设计与数据层**
  - 数据存储：用户数据、对话记录、知识库文档的存取与权限边界
  - 任务队列/异步：耗时任务的排队与进度反馈（避免“卡死”体验）
  - 成本控制：缓存、摘要、分层调用（轻量模型/重模型策略）
- **工程质量与可维护性**
  - 模块化与可配置：把 prompt/参数/模型选择做成配置而非硬编码
  - 可观测性：日志、指标、追踪（能定位“为什么回答变差”）
  - 安全与合规：权限校验、敏感信息过滤、提示词注入对抗的工程化策略
- **部署与运维（可交付）**
  - Docker 容器化、环境隔离、配置管理
  - 基础 CI（可选）：最少包含 lint/test/build（能让队友放心合并）
- **阶段产出物**
  - 《稳定运行的线上/云端服务》（可访问、可演示）
  - 《接口文档》（OpenAPI/Swagger 或清晰的 API 说明）
  - 《RAG/评测报告》（含数据来源、评测集、指标与结论）
  - 《性能与成本报告》（响应时间、并发能力、费用估算与优化策略）

## 7. 黑客松（实战）里程碑（可嵌入深耕期中的关键检查点）

> 若仍保留 48-72 小时短时黑客松，可作为深耕期中的“期中大考/期末大考”。

### Milestone 1：需求评审（Requirement Review）

- 市场：用证据链宣讲需求（1 页 PRD 简报）
- 产品：低保真线框图（核心流程 + 关键状态）
- 技术：技术可行性评估（成本/风险/时间）
- 规则：需求可被“驳回/降级/冻结”，模拟企业评审

### Milestone 2：MVP 开发（Sprint）

- 市场：路演材料框架 + 种子用户访谈计划
- 产品：高保真 UI + AI 交互脚本 + Prompt 版本迭代
- 技术：核心功能实现 + 接口对接 + 质量保障（日志/监控）

### Milestone 3：验收与交付（Delivery）

- 全员：联调测试、演示脚本、部署可用
- 市场作为 QA：对照需求验收，不仅找 bug，更找“体验断点”和“价值偏差”

## 8. 第三阶段：蜕变期 —— 职业化交付与人才落位（2-4 周）

### 8.1 阶段目标

按企业标准验收，完成从“学生作业”到“职业作品”的包装。  
关键词：标准化、复盘、自我营销、校企对接。

### 8.2 通识必修（全员）

- 职业伦理与法律：数据隐私、开源协议、AI 生成内容版权与标注
- 结构化表达：金字塔原理，5 分钟高效汇报（面向评委/高管）
- 项目复盘方法：What/So what/Now what；失败归因与行动项

### 8.3 分角色完赛技能树（市场 & 产品强化版）

#### 市场方向（终章：价值传递与策略）

- 商业路演（Pitching）：不仅讲功能，更讲愿景、规模、壁垒
- 竞品防御：当被问“大厂也做你怎么办”，用数据与垂直场景构建护城河
- 行业分析写作：把项目写成可传播的洞察文章（带方法论与证据）
- **最终交付物（简历亮点）**
  - 《行业深度洞察白皮书》（可公开传播）
  - 《商业计划书 BP + 证据附件》
  - 《增长实验与数据复盘包》

#### 产品方向（终章：作品集与反思）

- Case Study 写作：展示“发现问题-探索方案-选择权衡-验证迭代”
- 视频包装：30-60 秒产品演示短片（高燃但真实）
- 设计验收：像素级验收（Pixel Perfect）+ AI 输出一致性验收
- **最终交付物（简历亮点）**
  - 在线作品集 Portfolio（含完整推导过程与迭代证据）
  - 《AI 交互规范与提示词体系（开源版/脱敏版）》
  - 《可用性测试与改版记录》

#### 技术方向（终章：开源交付与工程表达）

- 技术文档写作：README、架构说明、部署指南、常见问题（企业非常看重）
- 代码重构与规范化：清理“屎山”、抽离通用模块、补齐注释与错误处理
- 测试与质量说明：最少可复现、关键路径测试、回归测试策略（含 prompt regression）
- 安全与合规收口：隐私处理、权限边界、依赖与开源协议说明
- 开源贡献与影响力：尝试把通用模块开源，或给 Datawhale/开源生态提交 PR
- **最终交付物（简历亮点）**
  - GitHub 开源项目（可复现、可部署、文档齐全）
  - 《技术架构与关键决策记录》（为什么这样选型/权衡）
  - 《线上 Demo 与监控截图/运行证据》（不是“只能在我电脑上跑”）

## 9. Datawhale 的桥梁作用：人才评估体系（企业可用）

我们不只发奖状，更发“能力评估报告”。建议给企业提供学生能力三维雷达图：

1) **硬技能（Hard Skills）**：调研深度、设计规范、交付质量、工具熟练度  
2) **软技能（Soft Skills）**：沟通协作、抗压、表达、复盘与推进能力  
3) **AI 协作力（AI Quotient）**：是否能把 AI 当作“工作流的一部分”提升效率与质量（含评测意识与风险控制）

## 10. 长期备赛模式的机制建议（防划水 + 真企业化）

1) **月度路演（Monthly Demo Day）**

- 每月一次阶段汇报，邀请企业导师参加
- 若 demo 无法运行或进度严重滞后：扣分或重组（模拟“熔断”）

2) **轮岗体验（Job Rotation）**

- 深耕期中期安排 1 周角色互换
- 目的：培养同理心，减少“技术看不起市场/产品逼死开发”的结构性冲突

3) **Mentor 制度（Office Hour）**

- 每 2 周固定答疑：技术问题找 Datawhale 技术导师；商业逻辑找企业 PM/运营
- 形成“问题 → 方案 → 决策 → 复盘”的闭环记录

---

## 11. 最终总结：三阶段能力进阶（长期版）

- **一阶段（装备）**：知道要做什么，掌握基础兵器（2 周）
- **二阶段（深耕）**：知道怎么做好，经历工程化与数据化迭代（4-8 周）
- **三阶段（蜕变）**：知道怎么卖好，把成果职业化，完成校企对接（2-4 周）

> 这套长期方案能最大化保证“完赛即就业”的转化率：既能交付作品，也能交付可被企业识别的能力证据链。
