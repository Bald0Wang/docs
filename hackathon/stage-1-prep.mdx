## 1. 通识必修（全员）

- **AI 认知对齐**：理解大模型的能力边界与幻觉问题。
- **协作工具链**：掌握 Feishu/Notion 协作及 Trello 看板任务拆解。
- **敏捷基础**：学会进行高效的 Stand-up Meeting（每日站会）。
- **MVP 思维**：学会砍需求（MoSCoW），把“想做”变成“能交付的一条闭环”。
- **决策可追溯**：所有关键取舍必须记录（为什么砍、砍到哪、风险是什么）。

### 每日站会模板（10–15 分钟）

- **我昨天完成了什么**（可验证：链接/截图/PR/文档段落）
- **我今天要完成什么**（可验收：明确 DoD）
- **我遇到的阻塞是什么**（需要谁/什么资源）
- **我对需求/范围的风险预警**（越早说越好）

## 2. 专精技能树（分角色任务书）

> 说明：这一阶段不追求“做得很精致”，追求“可评审、可落地、可协作”。

### 2.1 市场方向（文科类）

#### 核心任务
- **痛点挖掘**：区分“伪需求”与“真痛点”（5W1H + JTBD）。
- **竞品/替代方案侦察**：快速总结竞品优劣势（不要只列产品名，要列能力差异）。
- **证据链构建**：每个结论必须有证据（访谈摘录/公开数据/真实案例）。
- **方向验收口径**：提前定义“做成什么算成功”（最小可解释指标）。

#### 建议交付模板：问题定义一页纸（Problem Statement）

| 字段 | 填写说明 |
| :--- | :--- |
| 目标用户是谁 | 具体到"某类人 + 某个场景" |
| 用户在什么场景遇到什么痛 | 用可观察的行为描述，不用情绪词糊弄 |
| 现有替代方案是什么 | 人工/工具/竞品/流程 |
| 我们的方案一句话 | 价值主张（UVP） |
| 成功怎么衡量 | 1 个主指标 + 1 个辅指标（可解释、可测） |
| 风险与边界 | 不做什么、什么做不到、合规风险 |

#### 用户访谈最小打法（5 人也有效）
- 目标：验证“痛点是否真实、是否高频、是否愿意改变习惯”
- 输出：**3 条可引用原话** + **2 条反例**（不要只收集支持你的观点）

### 2.2 产品方向（设计类）

#### 核心任务
- **用户旅程**：绘制 User Journey Map（触点/情绪/阻塞/机会点）。
- **核心流程低保真**：先把主链路走通，再考虑美观。
- **关键状态清单**：空状态、异常状态、加载/超时、拒答与引导。
- **AI 交互基础**：定义 AI 在流程中的角色（生成/检索/判断/总结），明确输入输出。

#### 本阶段必须交付：核心流程线框图（低保真）
- 只画 **1 条主流程**：用户从触发 → 输入 → 系统处理 → 输出结果 → 形成价值
- 至少补齐 **3 类状态**：空状态 / 错误状态 / AI 不确定或拒答状态

#### AI 交互说明 v0（最小版本）

| 字段 | 示例/说明 |
| :--- | :--- |
| AI 的角色 | 助手 / 审核员 / 检索员 / 规划器 |
| 输入来源 | 用户输入 / 上传文件 / 知识库 / 表单 |
| 输出格式 | 结构化要素（表格/要点/步骤/引用） |
| 禁止事项 | 不编造引用；不输出隐私；不越权给建议 |
| 失败处理 | 给出澄清问题/引导用户补充信息 |

### 2.3 技术方向（理工类）

#### 核心任务
- **Git 工作流**：分支策略、PR、Code Review（避免“口头合并”）。
- **API 经济**：优先使用成熟 API/模型能力，避免“从头训练”的不切实际。
- **快速开发**：用 Streamlit/Gradio/最简单前后端做出可演示雏形。
- **可行性评估**：对产品方案做边界判断（时间/成本/风险），敢于说“不”。

#### 技术可行性评估 v0（模板）

| 字段 | 填写说明 |
| :--- | :--- |
| 核心链路 | 端到端必须跑通的最小链路 |
| 关键依赖 | 模型/接口/数据/第三方服务 |
| 成本估算 | 调用次数、费用区间、预算上限 |
| 风险清单 | 幻觉/合规/隐私/性能/稳定性 |
| 备选方案 | A 不行就用 B（降级策略） |
| DoD | 什么算“交付完成” |

## 3. Milestone 1：需求评审（Requirement Review）

> 阶段末必须过这一关：需求可被驳回/降级/冻结，确保后续可交付。

### 评审输入
- **市场**：问题定义一页纸 + 竞品矩阵 + 证据链（至少 3 条可引用证据）
- **产品**：核心流程低保真 + 关键状态清单 + AI 交互说明 v0
- **技术**：可行性评估 v0 + 成本/风险/备选方案 + MVP 交付承诺

### 评审输出
- **冻结后的 MVP 范围**（写入文档，成为后续开发依据）
- **互锁规则确认**（无证据不设计、无原型不开发、无验收不交付）
- **下一阶段 Sprint 0 任务列表**（进入看板）
